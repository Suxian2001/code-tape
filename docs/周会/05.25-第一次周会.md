## **3. 技术路线与 ADR 依据**

### **3.1 事件流优先**

PRD 明确要求录制结果不是传统视频，而是带时间戳的操作事件流。当前事件模型围绕以下事件组织：

* `content-change`：内容变更，P0 保存变更后的完整代码。

* `selection-change`：光标和选区变化。

* `mouse-move`：鼠标轨迹，需采样降频。

* `shortcut`：快捷键触发。

* `run-start` / `run-output` / `run-error`：运行开始、输出、错误。

* `record-start` / `record-pause` / `record-resume` / `record-stop`：录制状态。

* `media-toggle` / `media-warning`：媒体能力和异常。

完整代码快照会牺牲体积，但能换来 P0 的稳定性：

* seek 更容易恢复。

* 回放状态更容易验证。

* 不需要过早处理 diff、operation log、CRDT/OT。

* 更符合 PRD 中“记录变更后的完整代码内容”的建议。

导师可重点判断：**完整代码快照是否会在演示规模下带来不可接受的体积问题？** &#x20;

当前 ADR-025 https://github.com/ceilf6/code-tape/discussions/52 已通过预算把演示文件规模限制在可控范围内。



### **5.2 录制包优先本地闭环**

当前录制包路线：

1. 录制过程中收集事件和媒体。

2. 生成 meta、events、snapshots、media、manifest。

3) 保存到 IndexedDB。

4) 使用 manifest status 做两阶段提交，避免刷新或 quota 中断产生半成品。

5. 加载时校验 schemaVersion、checksum 和 manifest。

6. 失败时通过 zip 导出/导入兜底。

这个路线的优点：

* 不依赖后端，减少 P0 外部变量。

* 能支撑本地录制中心。

* 与 P1 云端模型兼容。

* 便于 Demo 准备备用录制包。

风险：

* IndexedDB quota 受浏览器和磁盘情况影响。

* Blob 与 JSON 的一致性需要靠 manifest 和 checksum 约束。

* 导入导出 zip 需要严格校验，避免非法包污染列表。

需要导师判断：**P0 是否必须把 zip 导入导出做成显性功能，还是只作为兜底路径？**



### **5.3 回放 seek 路线**

当前回放 seek 采用：

> 最近 inclusive 快照 + 增量事件静默重放。

含义是：

* 快照中的 `eventSeq` 已经包含所有 `seq <= eventSeq` 的事件。

* seek 到目标时间时，先找到最近的快照。

* 再按 `(timestampMs, seq)` 应用快照之后到目标时间之间的事件。

* 所有 seek 入口必须统一走 `ReplayScheduler.seek(targetTimeMs)`。

这个方案比“每次从头重放”更适合长录制，也比“每个时间点保存全量状态”更节省空间。

风险：

* 快照生成时机要稳定。

* 短生命周期 UI，例如快捷键浮层，需要 seek 后重新计算，而不是直接套用过去动画状态。

* 媒体 currentTime 和事件 timestampMs 需要映射，否则容易 drift。

希望导师确认：**这套 seek 策略是否足以支撑 P0，是否需要在 Demo 前增加更强的 seek 压测？**



### **5.4 媒体同步路线**

当前媒体路线：

* 麦克风 + 摄像头合并为单 WebM Blob。

* 录制前选择设备，录制中只开关当前 track，不切换物理设备。

* 事件流 timestampMs 是唯一事实时间轴。

* 媒体通过 MediaClockAdapter 映射到 timeline time。

* 正常播放时听感优先，但 drift 超阈值需要纠偏。

这样做的原因：

* 单 WebM 最容易和播放器控制条结合。

* P0 不引入分段媒体模型。

* 为 P1 分段媒体留接口，不在 P0 实现复杂切换。

风险：

* MediaRecorder 和 WebM 在不同浏览器差异较大。

* Safari / Firefox 不是主演示路径。

* 如果媒体 stalled/buffering，需要暂停事件推进或允许纯事件流回放。



### **5.5 iframe sandbox 路线**

当前代码运行路线：

* P0 使用 iframe sandbox。

* 每次运行创建全新 iframe。

* 使用随机 runId、postMessage 协议、event.source 校验、runId 校验、schema 校验。

* 设置 3s timeout。

* JS 运行是 P0 必达；TS 简单运行可选。

* 回放时默认展示历史运行结果，不自动重新执行。

优点：

* 不需要后端服务。

* 能满足 PRD 的“前端代码页面展示”二选一。

* 对 Demo 成本最低。

* 运行输出能进入事件流。

风险：

* 不能支持 npm、Node API、多语言后端执行。

* sandbox iframe 的 postMessage 需要严格校验。

* 复杂 TS 或依赖型代码不适合 P0 演示。

希望导师确认：**如果评审问“为什么不做后端代码沙箱”，是否可以用 PRD 二选一和 P0 风险收敛作为解释？**



## **6. Demo 验收与展示脚本**

### **6.1 建议展示重点**

* seek 状态恢复。

不建议把主要时间花在：

* 页面视觉装饰。

* 完整 IDE 功能。

* 多语言执行。

* 云端登录分享。

* WebRTC 实时通话。

* AI 字幕生成。

### **6.2 建议 Demo 脚本**

建议准备一段固定代码讲解，例如：

1. 写一个 `debounce` 或 `twoSum`。

2. 先讲问题，再写初版代码。

3) 选中一段代码解释。

4) 使用快捷键格式化或注释。

5. 点击运行，展示 stdout 或页面结果。

6. 修改一个 bug，再次运行。

7) 暂停录制，再继续。

8) 结束录制并保存。

9. 从回放中心进入回放。

10. seek 到 bug 修复前后，展示状态恢复。

11) 切换倍速，展示快捷键浮层和鼠标轨迹。

### **6.3 兜底策略**

建议至少准备：

* 一个现场录制演示。

* 一个提前录好的稳定录制包。

* 一个无媒体或媒体失败时的纯事件流录制包。

* 一个短版本 Demo 脚本，控制在 3 分钟内。

* 一个长版本 Demo 脚本，控制在 5 分钟内。

如果现场权限或媒体失败：

* 继续展示编辑器事件流、运行结果和回放 seek。

* 将媒体失败解释为浏览器权限/设备问题，并用备用包展示媒体同步。

如果 iframe runtime 失败：

* 展示 run-error 也会被记录进事件流。

* 使用备用包展示成功运行输出。

如果 IndexedDB 保存失败：

* 展示 zip 导出兜底。

* 用备用包导入回放。

## **7. 加分项取舍**

### **7.1 推荐优先级**

当前我建议加分项按如下顺序考虑：

1. Python 高亮、语言切换、录制语言状态、回放恢复。

2. AI 字幕基础 UI 或模拟字幕回放。

3) WebRTC 实时事件同步 PoC。

但只有当 P0 主链路稳定后，才进入加分项实现。

### **7.2 为什么 Python 高亮优先**

Python 高亮的优势：

* 实现成本低。

* 不要求执行。

* 能体现 schema 和编辑器语言扩展能力。

* 对主链路侵入小。

* Demo 中容易解释为“多语言录制能力的雏形”。

边界要说清楚：

* 只做高亮和状态回放。

* 不做 Python 执行。

* 不做包管理。

* 不做后端沙箱。

### **7.3 为什么 AI 字幕不建议抢 P0**

AI 字幕很有展示价值，但风险包括：

* 音频提取和格式处理。

* ASR 模型接入。

* 字幕时间戳对齐。

* 变量名、专业术语纠错。

* 性能和浏览器兼容。

* 字幕与 seek 的双向同步。

建议 P1+ 分级实现：

* 第一级：带时间戳字幕、滚动展示、点击 seek。

* 第二级：术语/变量名纠错、自动分段、章节生成。

### **7.4 为什么 WebRTC 不建议抢 P0**

WebRTC 实时面试很符合应用场景，但 P0 不建议投入，因为它会引入：

* 实时事件同步。

* 网络抖动和乱序。

* 断线重连。

* 多用户身份。

* 双向音视频通话。

* 面试结束后保存为回放。

这些问题都不应该挡住单人录制回放主链路。

## **8. 团队分工与推进节奏**

### **8.1 当前分工观察**

当前分工有一个优点：各 issue 都有清晰边界，能并行推进。

但需要注意两个集成风险：

1\. **录制链路集成风险**：RecorderControls、RuntimeProducer、CameraPreview、RecordingController、EventBus、PackageBuilder 最终都要在 RecorderPage 汇合。

如果每个模块都各自完成后才联调，可能会在 5/29 后集中爆雷。

## **9. 风险清单与导师问题**

### **9.1 高优先级风险**

| 风险                      | 影响            | 当前缓解方式                         | 希望导师判断          |
| ----------------------- | ------------- | ------------------------------ | --------------- |
| P0 范围继续膨胀               | DDL 前主链路不稳定   | ADR 明确延后项                      | 是否还要继续砍范围       |
| seek 状态恢复不稳定            | 回放核心价值受损      | inclusive 快照 + 增量静默重放          | 是否需要提前做专项压测     |
| 媒体同步复杂                  | 音视频和事件流错位     | 单 WebM + MediaClockAdapter     | 是否允许媒体失败时用备用包展示 |
| iframe runtime 安全边界不足   | 运行功能不可靠或有安全风险 | source/runId/schema/timeout 校验 | 是否还需补充安全限制      |
| IndexedDB quota 或导入导出失败 | 录制结果无法稳定保存    | manifest/checksum/zip 兜底       | zip 是否必须展示      |
| 端到端联调太晚                 | 模块完成但主链路不通    | 建议每日集成检查                       | 是否指定集成负责人       |

### **9.2 会上想问导师的问题**

按优先级排序：

1. 当前 P0 范围是否合理？有没有应该立即砍掉或必须补上的能力？

2. 事件流优先的产品表达是否准确？最终 Demo 是否应该突出“结构化时间线”而不是“在线 IDE”？

3) iframe sandbox 作为 P0 代码运行路线是否能通过验收？如果被问到不做后端沙箱，解释是否充分？

4) 完整代码快照作为 P0 content-change 事件是否可以接受？是否担心体积问题？

5. seek 的“inclusive 快照 + 增量事件静默重放”是否是合适方案？

6. 媒体同步是否必须现场实时演示，还是可以用备用包兜底？

7) P0 演示环境锁定 Chrome/Edge latest 是否可接受？

8) Python 高亮是否是最合适的第一加分项？AI 字幕和 WebRTC 是否继续延后？

9. 团队是否需要更早端到端联调，还是继续按模块 issue 并行？

10. 最终答辩时，导师建议我们用什么方式解释“这不是录屏工具”的差异化价值？
