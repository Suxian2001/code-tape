# 代码讲解工具一个月时间规划

> **阅读摘要**
>
> 本文基于 [技术模块拆解](技术模块拆解.md) 制定一个月实施计划。项目目标不是只完成 P0，而是在一个月 DDL 内同时完成 **P0 主流程** 与 **可展示的加分项**。P0 以 JavaScript / TypeScript 为基本语言范围，其他语言支持作为加分项处理。整体推进方式采用「先技术方案和高风险 PoC，再 P0 主链路实现，同时并行孵化加分项」。

## 一、目标与范围

### 1. 时间目标

以 2026-05-22 启动计算，一个月 DDL 对应 2026-06-22 前交付可演示版本。

如果实际评审时间有调整，排期比例保持不变：

- 前 20% 时间用于技术方案、关键选型和高风险 PoC。
- 中间 55% 时间用于 P0 录制与回放主链路。
- 后 15% 时间用于加分项接入和体验补齐。
- 最后 10% 时间用于联调、冻结、演示准备和兜底。

### 2. P0 必达范围

P0 必须围绕 [技术模块拆解](技术模块拆解.md) 中的五个主流程模块完成闭环：

| 模块 | P0 要求 |
| --- | --- |
| 代码编辑器模块 | 支持 JS / TS 编辑体验、语法高亮、基础补全、工具栏、代码运行入口 |
| 操作录制模块 | 采集内容变更、光标选区、鼠标轨迹、快捷键，并生成带时间戳事件流 |
| 音视频录制模块 | 采集麦克风、摄像头，支持开关、设备管理和摄像头预览 |
| 录制总控模块 | 支持开始、暂停、继续、结束录制，生成统一录制包 |
| 回放模块 | 支持播放、暂停、倍速、seek、音量控制，并同步还原编辑器、鼠标、选区、快捷键和摄像头 |

P0 的演示验收口径：

1. 用户可以在编辑器中完成一段 JS / TS 代码讲解。
2. 系统可以录制编辑器内容变化、光标选区、鼠标轨迹、快捷键、麦克风和摄像头。
3. 录制结束后可以生成录制包。
4. 回放时可以按时间轴还原讲解过程。
5. 回放支持暂停、继续、倍速、进度条 seek、音量调节和静音。

### 3. 加分项范围

加分项分为「必争」和「冲刺」两层，避免全部压到最后一周。

| 加分项 | 优先级 | 目标 |
| --- | --- | --- |
| 其他语言支持 | 必争 | 在 JS / TS 基础上扩展其他语言的高亮和语言切换 |
| AI 字幕基础版 | 必争 | 从录制音频生成带时间戳字幕，并在回放时展示 |
| WebRTC 实时面试最小链路 | 冲刺 | 实时同步编辑器事件，条件允许时接入双向音视频 |

其他语言支持作为加分项，不影响 JS / TS 作为 P0 基本项的交付。

## 二、核心推进原则

### 1. 主链路优先

技术主线必须始终围绕下面这条链路推进：

```text
编辑器集成
-> 操作事件采集
-> 音视频采集
-> 录制总控打包
-> 回放时间轴调度
-> 同步展示
```

所有加分项都必须依附这条链路，不能反向拖慢 P0。

### 2. 加分项提前留口

加分项不能等 P0 全部完成后才开始。第一阶段就需要在技术方案中预留：

- 事件模型中的 `language`、`timestamp`、`track`、`source` 等扩展字段。
- 录制包中的字幕、章节、多语言元数据扩展位。
- 回放模块中的字幕层、实时事件输入层。
- 编辑器模块中的语言注册和切换机制。

### 3. 技术方案阶段敲定未决问题

以下问题不在本文中直接替项目拍板，需要在技术方案阶段明确：

- 代码执行路线选择后端代码沙箱，还是前端页面展示。
- TypeScript 是 P0 编译执行，还是 P0 只要求编辑体验和展示。
- 音频和摄像头是否合并为同一个媒体文件。
- P0 录制包只做本地保存，还是必须接入上传。
- 回放时除 JS / TS 外，第一批加分语言支持哪些。

## 七、阶段四：加分项接入与体验补齐

### 1. 阶段目标

阶段四要在 P0 主链路可用的基础上，让项目具备更强展示效果。加分项要优先选择和主链路复用度高、演示效果明显的内容。

### 2. 加分项推荐顺序

| 顺序 | 加分项 | 原因 |
| --- | --- | --- |
| 1 | 其他语言支持 | 基于编辑器语言配置扩展，成本相对可控，展示效果直接 |
| 2 | AI 字幕基础版 | 与录制音频和回放时间轴强相关，产品完成度高 |
| 3 | WebRTC 实时面试最小链路 | 技术亮点强，但网络同步和音视频通话风险更高 |

### 3. 推荐 Issue 拆分

| Issue 标题 | 类型 | 建议分数 | 交付物 |
| --- | --- | --- | --- |
| `[加分项] 其他语言高亮与语言切换扩展` | 必争 | `score:3` | 除 JS / TS 外至少支持 1-2 种语言高亮 |
| `[加分项] 多语言录制包元数据支持` | 必争 | `score:2` | 录制包记录语言信息，回放时恢复语言状态 |
| `[加分项] AI 字幕生成基础链路` | 必争 | `score:5` | 从音频生成字幕文件或接入模拟字幕数据 |
| `[加分项] 字幕回放与点击 seek` | 必争 | `score:5` | 字幕随播放进度滚动，点击字幕跳转 |
| `[加分项] WebRTC 实时事件同步最小链路` | 冲刺 | `score:5` | 被面试者事件实时同步到面试官端 |
| `[加分项] WebRTC 双向音视频通话` | 冲刺 | `score:5` | 条件允许时接入音视频通话 |
| `[P1] UI 状态反馈与主题切换` | 体验补齐 | `score:3` | 深浅主题、录制/暂停/保存/回放状态反馈 |
| `[P1] 回放展示开关` | 体验补齐 | `score:2` | 快捷键、鼠标、字幕等展示开关 |

### 4. 加分项验收标准

必争加分项建议至少满足：

- 除 JS / TS 外，至少一种语言可以切换并正确高亮。
- 回放时可以恢复录制时的语言状态。
- 字幕可以在回放页面展示。
- 字幕和回放时间轴有基本同步关系。

冲刺加分项建议至少满足：

- 可以区分被面试者端和面试官端。
- 被面试者端编辑器事件可以实时同步到面试官端。
- 如果双向音视频来不及完成，至少保留实时事件同步作为可展示亮点。

| 风险 | 影响 | 兜底策略 |
| --- | --- | --- |
| seek 状态恢复复杂 | 回放体验不稳定 | 第一阶段必须做 PoC；实现时优先支持最近状态恢复 |
| 音视频与事件流不同步 | 讲解回放观感差 | 统一使用录制起点时间轴，媒体和事件都记录相对时间 |
| TypeScript 执行成本高 | 拖慢 P0 | 技术方案阶段明确是否只做 TS 编辑体验；执行能力可降级 |
| 后端沙箱安全复杂 | 实现风险高 | 可先采用前端 iframe 展示或受限执行作为 P0 路线 |
| WebRTC 风险高 | 加分项不可控 | 降级为实时事件同步，不强求完整双向音视频 |
| AI 字幕模型接入不稳定 | 加分项延期 | 允许先用模拟字幕文件打通字幕回放和 seek |
| 最后一周新增功能过多 | Demo 不稳定 | 6/18 后冻结新增能力，只做联调和演示打磨 |
